### Sandbox capability classification

The original `workspace-write` failures observed in tests 2–4 are caused by
`bubblewrap` (`bwrap`) being unable to bring up a loopback interface in its
network namespace. This requires `CAP_NET_ADMIN` and a working user namespace
on the host. The observed failure mode appears identically on:

- Linux x64 container host (`<platform-device>`).
- Android arm64 / Termux (Android does not expose the same namespace API).

It is therefore a **host-level capability constraint** that affects any
process invoking `bwrap` with network-namespacing on those hosts, not a
`codex-vl-exec` regression. The `codex-vl-exec` runtime itself executes
correctly when invoked under a sandbox policy that does not require the
unavailable host capability (`danger-full-access`, and on a host with the
required capability, also `workspace-write` and `read-only`).
